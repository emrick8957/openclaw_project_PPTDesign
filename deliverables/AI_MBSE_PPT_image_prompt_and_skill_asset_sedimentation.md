# AI_MBSE 页面图片评审后的 Skill 资产沉淀建议与高质量生成提示词

## 一、哪些优化可以沉淀到 huawei_ppt_master Skill 资产里

### 1. 建议沉淀到 `templates/visual_rules.md`

#### 1.1 红色锚点不是看面积，而是看注意力分布

当前 P1-P7 红色像素占比都低于 1.1%，但仍出现“红色锚点分散”。因此建议新增规则：

- 红色面积低不等于视觉克制；
- 单页红色只能承担一种语义：标题关键词 / 主结论 / 关键风险 / 核心节点四选一；
- 禁止同页同时出现：红色标题关键词、红色编号、红色箭头、红色底部结论条、红色 bullet 多点抢焦点；
- 图片生成后必须检查“红色注意力点数量”，而不是只检查红色面积。

#### 1.2 卡片要有工程化层级，避免咨询模板感

建议新增：

- 多卡片页必须区分“主卡 / 支撑卡 / 注释卡”；
- 主卡最多 1 张，支撑卡 2~4 张，注释卡只能用灰边/无底色；
- 避免大圆角、浅粉/浅绿柔和底色、轻阴影叠加形成“咨询海报感”；
- 工程汇报优先使用白底、灰边、细线、明确对齐，不依赖装饰性卡片。

#### 1.3 底部结论条必须提供增量

建议新增：

- 底部结论条不能重复标题；
- 底部结论条应承担“因此 / 所以 / 下一步”的收口；
- 普通正文页底部结论条只用浅灰底或窄红线；
- 强收口页才允许更强红色结论框。

### 2. 建议沉淀到 `visual_patterns/layout_library.md`

#### 2.1 `four_quadrant_judgement` 增加坐标有效性约束

P4 暴露的问题是：四象限形式存在，但坐标语义不成立。建议补充：

- 四象限必须有真实 x/y 轴含义；
- 每个象限的位置必须能被坐标解释；
- 如果只是四类概念并列，不得使用四象限，优先改用 `three_column_cards`、`multi_module_solution_overview` 或 `layered_architecture`；
- 四象限必须标出目标区 / 风险区 / 过渡区，否则容易变成装饰。

#### 2.2 `stack_architecture_with_right_insights` 增加“主图—洞察”关系约束

P5 暴露的问题是右侧洞察像备注。建议补充：

- 右侧洞察栏最多 1 句主结论 + 3 条短要点；
- 右侧洞察必须解释主图，而不能另讲一套；
- 主图必须表达因果 / 层级 / 流向，不能只是节点堆叠；
- 若页面核心矛盾是 A≠B，必须在主图中显式表达。

#### 2.3 `three_column_cards` 增加“并列 vs 链路”判断

P6 暴露的问题是三栏清楚但没有表达递归放大。建议补充：

- 如果三栏是并列分类，可使用 `three_column_cards`；
- 如果三栏存在先后、因果、递归、传导关系，应增加箭头主线，或改用 `swimlane_process` / `value_chain_loop`；
- 三栏页每栏内部标签建议统一为：缺口 / 影响 / 治理，避免“表现/后果/抓手”过泛。

#### 2.4 `two_column_comparison` 增加“表格转面板”规则

P7 暴露的问题是表格正确但演示张力不足。建议补充：

- 高层汇报中的两列对比优先做“左右对比面板”，而非 Excel 式大表；
- 左右两列必须有明确差异结论；
- 右侧目标态/新范式列可用标签化关键词强化，不只靠红字；
- 底部必须有“因此”类结论。

---

### 3. 建议沉淀到 `templates/chart_patterns.md`

#### 3.1 `quadrant_matrix` 增加失败条件

- 如果 x/y 轴不能解释每个象限，不得使用；
- 如果只是四类能力或四类问题，不得伪装成 quadrant；
- 必须声明：风险区、目标区、优先区或迁移路径。

#### 3.2 `architecture_flow_diagram` 增加因果链要求

- 流程图必须表达输入、处理、输出、缺口或判断；
- 如果页面核心是“能判断 ≠ 能解释”，流程图必须把“判断能力”和“证据解释能力”拆成两个节点；
- 不允许把不同层级概念放成同级节点。

#### 3.3 `gap_heatmap` 增加与 `three_column_cards` 的边界

- 如果只有三类缺口，并没有热度/严重度/优先级矩阵，不应使用 `gap_heatmap`；
- 可使用 `comparison_table` 或 `three_column_cards`；
- 如果要表达风险传导，优先使用 `swimlane_process` 或 `value_chain_loop`。

---

### 4. 建议沉淀到 `templates/wording_rules.md`

#### 4.1 强判断标题必须配强支撑

建议新增规则：

- 标题出现“不是 A，而是 B”时，正文必须按“现状—误区—风险—行动”展开；
- 标题出现“倒逼 / 重构 / 转向”时，正文必须说明驱动力、变化对象、治理动作；
- 标题出现“无法支撑 / 不可信 / 风险放大”时，正文必须补齐证据链或机制链。

#### 4.2 抽象概念必须给一句定义

例如：

- `AI-ready MBSE = 可被 AI 读取、验证、追溯并受控参与的工程模型体系`；
- `可信工程底座 = 来源、推导、状态、责任、审查可追溯的模型记录`；
- `贡献治理 = AI 内容先候选、再审查、后提交、可回溯`。

---

### 5. 建议沉淀到 `eval/visual_scorecard.md`

#### 5.1 新增评分项：图表是否把结论讲透

建议新增 10 分项：

- 坐标 / 箭头 / 层级关系是否成立；
- 图表是否表达因果，而非只摆放概念；
- 页面核心矛盾是否被视觉化；
- 底部结论是否提供增量。

#### 5.2 新增一票降级项

- 四象限坐标语义不成立：最高 B 档；
- 页面核心矛盾没有在图中体现：最高 B 档；
- 右侧洞察栏变成备注列表：最高 B 档；
- 三栏卡片存在传导关系但没有箭头/流程主线：最高 B 档。

## 二、生成 PPT 图片的高质量提示词

### 提示词 1：通用高质量页面图片生成提示词

```text
请基于当前 deck_spec / 页面设计说明生成华为风格 PPT 页面图片。

必须遵守以下规则：

1. 页面定位
- 16:9 横版，白底为主，红黑灰白配色；
- 风格为华为高层汇报，不要互联网发布会风、咨询海报风、卡通插画风；
- 每页只服务一个核心判断，标题必须结论先行。

2. 页面结构
- 标题区占页面上方 10%~15%，标题 1~2 行，不能压缩主体；
- 主体区占 75%~80%，必须有清晰网格、对齐和模块层级；
- 页脚区约 5%，保留 Huawei Confidential / 来源 / 页码，小字浅灰，不抢主体；
- 正文页不要满屏文字，不要把所有信息塞进表格。

3. 红色使用
- 单页只允许 1 个主红色视觉锚点，最多 2 个辅助红色强调；
- 红色只能用于：标题关键词 / 主结论 / 关键风险 / 核心节点，四选一为主；
- 不要同时使用红色标题、红色编号、红色箭头、红色卡片、红色底部条；
- 普通正文页红色面积控制在 5%以内，且注意力点必须集中。

4. 卡片与组件
- 卡片默认白底或浅灰底，细灰边，无重阴影；
- 强卡片最多 1 张，中卡片 2~4 张，弱卡片只用灰边或无底色；
- 避免大圆角、浅粉/浅绿大面积底色和多层框中框；
- 同页卡片圆角、边框、内边距、标题样式必须统一。

5. 图表语义
- 图表必须服务标题结论，不做装饰；
- 四象限必须有真实 x/y 轴逻辑，否则改成对比/分层/流程；
- 流程图必须表达输入、处理、输出、缺口或治理动作；
- 三栏卡片如果存在因果/传导关系，必须用箭头或流程线串起来；
- 对比页必须有明确差异结论，不要只是表格罗列。

6. 右侧洞察栏
- 右侧洞察栏最多 1 句结论 + 3 条短要点；
- 洞察栏必须解释主图，不得变成备注列表；
- 洞察栏宽度控制在页面宽度 25%~32%。

7. 底部结论
- 底部结论条必须提供增量，不能重复标题；
- 普通页使用窄红线、浅灰底或灰底红色关键词；
- 不要使用厚重红色压底条，除非是章节强收口页。

8. 输出质量
- 文字必须清晰可读，正文不小于 14pt；
- 图表元素对齐，不贴边，保留左右安全区；
- 页面应显得克制、工程化、有秩序，而不是模板堆砌。
```

### 提示词 2：基于单页 deck_spec 生成页面图片

```text
请根据以下单页 deck_spec 生成一张华为风格 PPT 页面图片。

输入：
- slide_no: {{slide_no}}
- title: {{title}}
- conclusion: {{conclusion}}
- display_text: {{display_text}}
- chart_type: {{chart_type}}
- chart_data: {{chart_data}}
- layout_pattern: {{layout_pattern}}
- visual_notes: {{visual_notes}}
- visual_focus: {{visual_focus}}
- must_highlight: {{must_highlight}}
- data_gaps: {{data_gaps}}

生成要求：
1. 严格匹配 layout_pattern，不得擅自改成其他页面结构；
2. 严格使用 chart_type 表达主图表，图表必须支撑 title 的核心判断；
3. 如果 chart_type 与内容不适配，优先保持 deck_spec 字段不变，但通过 visual_notes 的方式做最接近表达，不要自造新图表类型；
4. 页面必须包含：结论型标题、主图表/主结构、必要说明、页脚；
5. 每页只允许一个主红色锚点，红色语义必须明确；
6. 如果页面有卡片，必须体现强/中/弱层级；
7. 如果页面有流程、箭头或闭环，必须保证方向清晰、节点文字短、线条统一；
8. 如果页面有对比，必须让差异结论一眼可见；
9. 底部结论条不能重复标题，必须给出“因此/所以/下一步”的增量收口；
10. 输出前自检：
   - 标题是否一眼能看懂；
   - 主图是否把结论讲透；
   - 红色是否只有一个主焦点；
   - 卡片是否过多、过软、过装饰；
   - 页脚是否弱化且统一。
```

### 提示词 3：P4 这类“四象限/结构误区页”的专用提示词

```text
请生成一页华为风格 PPT，用于表达“工具能力增强不等于工程底座可信”。

核心结论：工具是放大器，不是替代品；AI 可用取决于语言、证据、治理是否同步补齐。

页面要求：
1. 不要做装饰性四象限；如果使用四象限，必须明确 x/y 轴含义，并标出风险区和目标区；
2. 更推荐使用“上层工具能力 + 下层可信底座”的分层结构：
   - 上层：RAG、插件、Agent 工作流；
   - 下层：语言表达、证据链、方法论治理；
   - 右侧：结构性误区与正确判断；
3. 红色只强调“工具增强 ≠ 可靠推理”；
4. 页面必须让读者一眼看到：当前误区是过度强化工具，真正短板是底座和治理；
5. 不要使用多种柔和底色，统一白底、灰线、少量华为红。
```

### 提示词 4：P5 这类“案例链路/能判断但不能解释页”的专用提示词

```text
请生成一页华为风格 PPT，用于表达“模型合规不等于证据闭环，AI 可判断但难解释为什么”。

页面主线：
模型给出什么 → AI 能判断什么 → 工程记录缺什么 → 因此不能支撑 why 解释。

页面要求：
1. 左侧/中部使用三段因果链：
   - 模型输入：100 km/h、10 m、poor road、satisfy；
   - AI判断：物理一致性可被识别；
   - 工程缺口：来源、推导、假设关联、验证状态缺失；
2. 右侧洞察栏只能放 1 句结论 + 3 个关键词：来源 / 推导 / 验证；
3. 必须显式表达“能判断 ≠ 能解释”；
4. 红色只用于“≠”或“证据闭环缺失”一个主焦点；
5. 底部结论：缺少工程证据链时，AI 答对也不能成为工程依据。
```

### 提示词 5：P6 这类“三类缺口/递归放大页”的专用提示词

```text
请生成一页华为风格 PPT，用于表达“AI参与MBSE的风险不是局部错误，而是跨消费、推理、回写的系统性放大”。

页面主线：
上下文缺失 → 证据状态缺失 → 贡献治理缺失 → 污染后续推理。

页面要求：
1. 不要只做三张并列卡片，必须用横向箭头或流程线表达递归放大；
2. 每个节点统一三类标签：
   - 缺口；
   - 影响；
   - 治理；
3. 红色只标“递归放大”或“污染回写”一个关键风险；
4. 减少内层小框，不做外卡片套内卡片；
5. 底部强结论：三类缺口会递归放大，必须通过状态治理与人工审查前置拦截。
```

### 提示词 6：生成后视觉自检提示词

```text
请对生成的 PPT 页面图片进行视觉自检，逐页输出评分和修改建议。

检查维度：
1. 与大纲/逐页文案/页面设计/deck_spec 的匹配度；
2. 与 visual_patterns/layout_library 的匹配度；
3. 与 templates/visual_rules 的匹配度；
4. 红色锚点是否唯一；
5. 卡片层级是否清晰；
6. 图表是否真正支撑页面结论；
7. 底部结论是否提供增量；
8. 页脚是否统一弱化；
9. 是否存在咨询模板感、互联网发布会风、装饰性图表；
10. 是否需要返工。

评分规则：
- 90分以上：可用；
- 80-89分：轻量优化；
- 70-79分：建议返工局部结构；
- 70分以下：必须重做。

一票降级项：
- 四象限坐标语义不成立；
- 页面核心矛盾没有视觉化；
- 红色出现 3 个以上抢焦点区域；
- 右侧洞察栏变成备注列表；
- 三栏卡片存在传导关系但没有箭头/流程主线；
- 底部结论条重复标题且无增量。
```
