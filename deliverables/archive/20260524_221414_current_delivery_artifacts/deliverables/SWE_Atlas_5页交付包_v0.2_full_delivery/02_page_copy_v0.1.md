# SWE Atlas：编码智能体基准评测核心内容与启示 - 逐页文案 v0.1

## P1

页标题：当前编码智能体的关键短板，不是“会不会写代码”，而是“能否按专业工程标准稳定交付”
一句话结论：SWE Atlas 证明，编码智能体评测需要从功能正确性扩展到工程质量、运行时证据和交付一致性。
正文模块：
- 论文提出 SWE Atlas，用于评测编码智能体在三类真实软件工程工作流中的能力：
  - Codebase Q&A：代码库理解与运行时问答，共 124 个任务；
  - Test Writing：测试编写，共 90 个任务；
  - Refactoring：重构，共 70 个任务。
- 数据集共 284 个专家编写任务，覆盖 18 个活跃开源仓库。
- 与传统 SWE-Bench 类评测不同，SWE Atlas 不只看 bug fix / feature implementation，而是评估“专业软件工程质量”：测试完整性、可维护性、抽象复用、代码库卫生、反模式等。
- 核心判断：头部模型在简单功能修复上接近饱和，但在复杂工程任务中仍不稳定。

图表内容：3类任务结构图 + “功能正确性 → 工程质量”评测升级箭头。
版式说明：顶部结论标题；左侧放“为什么需要新评测”；右侧放 3 个任务卡片；底部一句启示。
讲解口径：这篇论文的价值不是再给模型排一次名，而是把编码智能体的评价标准从“能跑通”提升到“像工程师一样可靠交付”。

## P2

页标题：SWE Atlas 的核心设计，是用“真实任务 + 混合评测”衡量智能体工程能力
一句话结论：SWE Atlas 将任务设计成低规格、强探索、需运行验证的工程场景，并用程序化检查与 Rubric 评审共同判定结果。
正文模块：
- **任务设计更贴近真实使用：**
  - Codebase Q&A 要求智能体探索代码库、搭建运行环境、追踪运行时行为，而不是只做静态代码阅读。
  - Test Writing 要求定位被测代码、补齐边界场景、提交新增测试 manifest。
  - Refactoring 要求在不改变行为的前提下完成模块边界设计、调用链传播、旧实现清理。
- **评测机制更完整：**
  - Codebase Q&A：Rubric 检查答案完整性和负向项。
  - Test Writing：Manifest check + Mutation check + Rubric check。
  - Refactoring：Regression tests + Rubric check。
- **Rubric 的作用：** 捕捉程序化测试难以覆盖的工程质量，如代码放置、可维护性、反模式、清理是否彻底。

图表内容：三泳道流程图：任务类型 / 评测机制 / 通过标准。
版式说明：中间用三列泳道分别展示 Q&A、测试、重构；每列下方放评测项；右侧放“为什么比单元测试更严格”。
讲解口径：SWE Atlas 的强点在于它承认软件工程不是单一 pass/fail 问题，工程质量需要多维度判定。

## P4

页标题：失败模式集中在三类：缺少运行时证据、测试断言弱、重构清理不彻底
一句话结论：当前智能体的主要问题不是完全不会做，而是经常“做得像”，但没有覆盖关键边界和工程闭环。
正文模块：
- **Codebase Q&A：缺运行时证据或信息不完整。**
  - 高分模型通常会更多执行代码、启动应用、发送请求并做运行时分析。
  - 失败常见于只做静态解释，或跑了实验但没覆盖所有必须回答的问题。
- **Test Writing：测试数量不等于测试质量。**
  - 模型常写 happy path 测试，断言较弱；测试能覆盖函数，但无法区分正常实现和 no-op mutation。
  - 真正有效的测试应覆盖边界、负向场景、不变量和可观察副作用。
- **Refactoring：行为保持容易过，工程质量难过。**
  - 仅用回归测试评估时，模型看起来更高分；加入 Rubric 后分数明显下降。
  - 多文件重构、调用点更新、旧代码清理、缓存/接口语义保持仍是明显短板。

图表内容：三个失败模式卡片：Evidence / Assertion / Cleanup，每张卡片给出“表现—原因—改进方向”。
版式说明：三列卡片，顶部用红色标签标注失败模式；底部给出“工程化评测必须看过程证据和交付质量”。
讲解口径：这页要强调：模型不是简单能力不足，而是工程闭环不完整；如果评测只看测试是否通过，会掩盖真实风险。

## P5

页标题：编码智能体建设应从“能力演示”转向“评测驱动的工程闭环”
一句话结论：若要把编码智能体用于真实研发，应建立覆盖任务、证据、质量和稳定性的内部评测体系，而不是只看 demo 成功率。
正文模块：
1. **评测对象要扩展：**
   - 不只评 bug fix，也要评代码库理解、测试生成、重构、运维排查、文档更新等专业工程任务。
2. **评测标准要分层：**
   - 功能正确性：测试是否通过；
   - 工程质量：可维护性、抽象、代码放置、死代码清理；
   - 过程证据：是否实际运行、是否覆盖边界、是否有日志/指标/调用链证明；
   - 稳定性：多次执行是否一致。
3. **测试生成要看“杀伤力”：**
   - 用 mutation / fault injection 检查测试是否能抓住错误，而不是只看新增测试数量。
4. **重构类任务要强制清理闭环：**
   - 新接口落地、旧接口删除、调用点覆盖、文档同步、反模式检查都应进入验收项。
5. **产品化建议：**
   - 建议内部构建 SWE Atlas 类评测集，按真实代码库沉淀任务与 Rubric，并接入 CI 做持续回归。
