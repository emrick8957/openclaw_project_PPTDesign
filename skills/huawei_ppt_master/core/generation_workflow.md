# 通用生成流程

## Step 1：需求澄清

在不影响执行的情况下，优先识别：
- PPT 主题；
- 汇报对象；
- 汇报类型；
- 页数范围；
- 输出阶段：大纲 / 逐页文案 / 页面设计 / PPTX 输入；
- 是否有模板、截图、PPT、文本材料、数据材料；
- 当前任务是从零生成，还是要求参考既有材料延续风格/结构/方法论。

如果用户没有给出汇报对象，默认按“高层汇报”处理。

## Step 1.5：参考材料识别与分类

如果用户提供了 PPT、截图、样张、文本材料或数据表，必须先判断：
1. 材料主要作用是风格参考、结构参考、论证参考，还是事实依据；
2. 材料更适合沉淀到视觉规则、版式规则、表达规则、方法论规则，还是主题知识包；
3. 材料是一次性参考，还是可进入长期规则库。

具体流程见：
- `core/reference_ingestion_workflow.md`
- `core/reference_material_policy.md`

## Step 2：主题路由

调用 `core/topic_router.md`，选择通用主题包或专属主题包。

主题包只能增强相关主题，不得污染无关主题。

## Step 3：叙事结构选择

根据汇报类型选择叙事结构：
- 战略规划：判断 → 背景 → 目标 → 路径 → 保障 → 决策；
- 技术方案：现状 → 问题 → 架构 → 能力 → 实施 → 运维；
- 客户方案：痛点 → 方案 → 价值 → 路径 → 保障 → 合作；
- 竞品对比：框架 → 维度 → 差距 → 机会 → 策略；
- 项目复盘：目标 → 进展 → 成果 → 问题 → 原因 → 改进；
- 运营治理：目标 → 机制 → 指标 → 闭环 → 风险 → 计划。

## Step 3.5：沉淀结果复用

若用户当前有明确指定偏好，以用户要求优先。

在正式生成大纲、逐页文案或页面设计前，应优先复用以下高置信规则：
1. 通用视觉规则；
2. 通用页面结构规则；
3. 通用表达规则；
4. 主题相关的方法论规则；
5. 已被验证且未冲突的主题知识包。

如果近期新增材料尚未完成冲突判断，宁可降级为一次性参考，也不要直接写入长期规则。

## Step 4：大纲生成

每页必须输出：
- 页码；
- 页标题；
- 核心结论；
- 页面类型；
- 建议图表；
- 页面设计说明；
- 该页回答的问题。

## Step 5：逐页文案生成

每页必须输出：
- 页标题；
- 一句话结论；
- 正文模块；
- 图表内容；
- 讲解口径；
- 数据缺口；
- 版式说明。

## Step 6：页面设计生成

从 `visual_patterns/layout_library.md` 选择版式模式，并说明：
- 页面区域划分；
- 标题区；
- 图表区；
- 文字区；
- 强调信息；
- 页脚；
- 可供 PPTX Builder 执行的布局说明。

同时必须输出视觉降噪约束：
- `red_anchor`：本页唯一主红色强调点；
- `card_hierarchy`：卡片强/中/弱层级安排；
- `spacing_rule`：模块间距、卡片内边距、页脚安全区；
- `bottom_bar_rule`：是否允许底部结论条，若允许需说明高度和颜色强度；
- `visual_simplification`：需要弱化或删除的装饰元素。
- `visual_boundary`：本页必须避免的负面视觉风格，如互联网发布会风、咨询海报风、卡通插画风、Excel 报表风。
- `page_type_gate`：本页页型门禁，如案例页不做图片墙、决策页拍板事项独立突出、路线图必须有阶段门槛、总结页必须形成最终定义。

页面设计必须遵守 `templates/visual_rules.md`、`visual_patterns/layout_library.md` 和 `templates/chart_patterns.md` 的视觉降噪规则。

## Step 6.5：deck_spec 字段映射检查
在输出 `deck_spec.json` 前，必须完成以下动作：
1. 先确定 `core_judgement`：本页唯一核心判断；
2. 再确定 `chart_proof_goal`：主图表必须证明什么；
3. 再确定 `chart_visual_boundary`：主图表视觉表达边界，建议 3~5 条；
4. 再判断该页主信息表达方式，确定 `chart_type`；
5. 最后判断页面区域组织方式，确定 `layout_pattern`；
6. 对照 `templates/chart_patterns.md` 检查 `chart_type` 是否为合法枚举；
7. 对照 `visual_patterns/layout_library.md` 检查 `layout_pattern` 是否为合法枚举；
8. 检查 `chart_type` 是否能证明 `chart_proof_goal`；
9. 检查 `chart_visual_boundary` 是否能约束该 `chart_type` 的视觉误用；
10. 若两者同名或字段越界，必须重写该页 deck_spec；
11. 若无精确匹配，按“最接近通用类型 + visual_notes 补充”的规则处理，不得自造新值。

## Step 7：自检

必须执行：
- 内容质量自检；
- 主题污染自检；
- 版式可执行性自检；
- 视觉降噪与华为风格一致性自检；
- 高层汇报价值自检；
- 参考材料使用边界自检；
- 学习沉淀冲突自检。
