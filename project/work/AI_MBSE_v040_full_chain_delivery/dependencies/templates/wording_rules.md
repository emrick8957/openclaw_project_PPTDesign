# 通用华为式文案规则

## 1. 总体风格

- 结论先行；
- 短句表达；
- 强判断；
- 少修饰；
- 多路径；
- 多抓手；
- 多闭环；
- 避免空泛口号。

## 2. 推荐词汇

抓手、牵引、底座、闭环、路径、机制、节奏、沉淀、复制、纳管、治理、运营、风险、边界、责任、里程碑、样板、标准化。

## 3. 禁止空话

不要写：

- 提升平台能力，赋能业务发展；
- 打造领先体系；
- 全面推进建设；
- 实现高质量发展；
- 构建完善生态。

应改成：

- 以统一机制为抓手，打通目标、资源、能力、运营四类关键链路。
- 当前核心矛盾不是单点能力不足，而是跨部门、跨系统、跨场景的闭环尚未打通。
- 建议采用“试点验证—能力沉淀—规模复制”的三阶段路径。

## 4. 标题规则

标题必须是结论型标题，而不是名词。

不要写：

- 建设背景
- 总体架构
- 风险分析
- 下一步计划

改成：

- 当前建设重点已从单点能力补齐转向体系化运营闭环
- 总体架构应围绕统一底座、能力复用和持续运营展开
- 主要风险集中在机制协同、数据质量和交付复制三个方面
- 下一阶段建议以样板场景牵引，先沉淀能力再规模复制

## 5. 主题专属表达

昇腾/NVIDIA、公安政务、AI平台、数据治理、网络安全等专属表达必须放在对应 `domain_profiles/` 中，不能写入全局文案规则。

## 6. 证据不足时的措辞规则
当用户未提供充分数据、公开研究或内部材料时：
- 可以输出判断框架、方向性结论、待验证假设；
- 不得把推测写成确定事实；
- 推荐使用：
  - 当前判断是……
  - 初步看……
  - 现阶段更可能的核心矛盾是……
  - 若后续材料验证成立，建议……
- 避免使用：
  - 已经证明……
  - 明确表明……（若无证据）
  - 核心原因就是……（若无证据）

## 7. 0517 解决方案型表达沉淀

来源：0517 PPT 图片。只沉淀标题与收口句式，不复用原文、具体品牌名、客户名、项目名或数据。

### 7.1 标题句式

- **趋势加速型**：`从{旧状态/单点能力}走向{新状态/体系化能力}`
- **能力牵引型**：`以{关键场景}牵引{平台/机制/能力}持续演进`
- **价值闭环型**：`围绕{目标对象}打通{能力—场景—运营—价值}闭环`
- **生态协同型**：`以{中心平台/核心能力}连接{伙伴/场景/行业}，形成可复制解决方案`
- **证明收口型**：`通过{样板/案例/指标}验证路径，再推动规模化复制`

### 7.2 页面收口规则

- 场景页收口：从“场景价值”落到“可复制能力”，避免只讲案例热闹。
- 能力页收口：从“能力清单”落到“建设抓手/运营机制”。
- 生态页收口：从“伙伴数量/覆盖范围”落到“协同关系和价值流”。
- 成果页收口：从“大数字”落到“下一步如何沉淀和复制”。

### 7.3 禁止泛化

- 不得把“赋能、打造、领先、全面升级”等词单独作为结论；必须补齐对象、抓手和动作。
- 不得无来源生成具体增长、覆盖、排名、案例数量等事实型表达。

## 8. 强判断标题的支撑规则

强判断标题必须配强支撑，避免“标题很硬、正文很散”。

### 8.1 “不是 A，而是 B”标题

正文必须按以下链路展开：

1. 现状：当前普遍做法或表象是什么；
2. 误区：为什么 A 不是核心；
3. 风险：继续停留在 A 会带来什么问题；
4. 行动：B 应如何建设或治理。

### 8.2 “倒逼 / 重构 / 转向”标题

正文必须说明：

- 驱动力是什么；
- 被改变的对象是什么；
- 重构后的目标态是什么；
- 需要什么治理动作。

### 8.3 “无法支撑 / 不可信 / 风险放大”标题

正文必须补齐证据链或机制链，至少回答：

- 缺什么；
- 为什么导致不可信；
- 影响哪个流程或决策；
- 应该用什么机制补齐。

## 9. 抽象概念定义规则

抽象概念首次出现时，必须给出一句可执行定义，避免口号化。

推荐句式：

- `{概念} = {对象} + {能力} + {治理边界}`；
- `{概念} 不是 {误区}，而是 {可执行目标}`；
- `{概念} 的最小闭环是 {输入}—{处理}—{审查}—{沉淀}`。

示例：

- `AI-ready 工程模型 = 可被 AI 读取、验证、追溯并受控参与的工程模型体系`；
- `可信工程底座 = 来源、推导、状态、责任、审查可追溯的工程记录`；
- `贡献治理 = AI 内容先候选、再审查、后提交、可回溯`。

## 10. 长标题压缩规则

当标题超过 28~32 个汉字且包含多个从句时，应优先压缩成强判断句，避免标题变成摘要段落。

推荐句式：

- `不是 A，而是 B`
- `核心不在 A，而在 B`
- `要先完成 A，再推进 B`
- `真正瓶颈不是 A，而是 B`
- `从 A 走向 B，关键在 C`

压缩后，副标题或讲解口径再承载背景说明。标题只承担一个判断，不同时承担背景、定义、路径和案例。

## 11. 总结页最终定义规则

总结页不应复述前文要点，而应形成最终定义或决策框架。

推荐句式：

- `X = 对象 + 能力 + 治理边界`
- `X 的本质不是 A，而是 B`
- `最终要形成 A、B、C、D 四类可验收能力`

如果总结页只罗列关键词，必须补一句“因此高层应如何判断/拍板/验收”的管理含义。


## 防骨架填词规则

禁止把固定句式作为字段生成器直接套用：

- 不得把 `core_judgement` 写成固定前缀 + `conclusion`；
- 不得把 `chart_proof_goal` 写成固定前缀 + 关键词拼接 + 固定后缀；
- 不得让 page_copy 的“核心判断”原样复制“一句话结论”；
- 不得让所有页面使用同一句讲解口径。

允许统一表达风格，也允许背景页对结论进行正当提炼，但每页必须形成独立带走点或图表证明关系。
