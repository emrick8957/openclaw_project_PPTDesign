# 生成PPT图片自我学习与迭代沉淀报告 v0.1

## 1. 本批 PPT 图片总体质量判断

本批 `AI4MBSE_自动化生产系统` 页面图片整体达到“可评审、可继续迭代”的华为风格基础水平：

- 具备结论先行意识，多数页面标题已是判断句；
- 白底、红黑灰、浅灰分区、页脚安全区基本稳定；
- 覆盖总起、目录、核心结论、问题定性、框架、能力地图、架构、案例、治理、路线图、决策、总结等完整页型；
- 图表方向基本匹配 deck_spec 中的 `layout_pattern` 与 `chart_type`；
- 未发现明显互联网发布会风、卡通插画风或大图堆砌风。

但与当前华为模板和已沉淀规则相比，仍处于“结构正确，但视觉语义和论证锋芒不足”的阶段。核心不是重新做主题内容，而是继续提升 **标题—图表—结论之间的论证闭环** 与 **组件层级克制**。

## 2. 与当前华为模板的主要差距

1. **模板更重“管理判断”，当前稿件仍有部分页面偏“概念说明”**  
   华为模板常在第一页或每页顶部直接给出管理结论；当前部分页面虽有判断，但仍需要阅读主体后才能理解重点。

2. **模板红色更像“定锚”，当前红色偶尔承担装饰/底座/强调多重职责**  
   P6 的大红底座是典型风险；其他页虽红色面积低，但主锚不总是唯一。

3. **模板卡片有主次，当前多卡片页仍有平均主义**  
   P2、P3、P12 等页的并列卡片整齐但不够有管理优先级。

4. **模板图表更强调因果、责任、路径、闭环，当前图表更多是结构展示**  
   P5、P8、P9 等页需要把结构图升级为“输入—过程—输出—验证/回流”的工程链条。

5. **模板案例页强调“证明一个判断”，当前案例页有素材墙风险**  
   P10 应弱化图片/模块并列，强化“路径可行”的证明链。

## 3. 高频问题 Top 10

1. 正文页红色底座或红色块权重过高，尤其 P6。
2. 多卡片同尺寸、同权重，缺少主卡/支撑卡/注释卡。
3. 部分标题偏长，结论句不够锋利。
4. 框架图展示组成多，展示机制少。
5. 右侧洞察区容易变成备注列表，而不是管理结论。
6. 表格页容易让表格吞没结论，缺少独立拍板项。
7. 路线图阶段并列清楚，但阶段门槛、验收产出不足。
8. 案例页图片/素材过多时，证明链条被削弱。
9. 总结页完成了关键词收敛，但最终定义感不足。
10. 页面之间节奏有波动：有些偏框架、有些偏说明、有些偏总结，章节角色可更稳定。

## 4. 可复用视觉规则

### 4.1 红色定锚规则增强

- 普通正文页不得让红色横条成为最大视觉块；大红条仅用于封面、章节、强收口页。
- 红色只能承担一种主语义：标题关键词、关键节点、风险点、最终结论四选一。
- 若页面已有红色标题关键词，则底部结论条优先使用浅灰底 + 红色关键词，不再使用厚重红底。

沉淀建议：通用级。

### 4.2 卡片层级规则增强

- 多卡片页必须显式指定：主卡 1 张、支撑卡 2~4 张、注释卡 0~2 张。
- 目录页、结论页、路线图页不能所有卡片等权；最后一格或主抓手必须成为视觉收口。
- 卡片统一样式不等于同等权重；可通过编号、标题字重、边框强弱和空间位置建立主次。

沉淀建议：通用级。

### 4.3 图表主角规则

- 中心平台、协同环境、治理机制、知识底座类页面必须让“中心主体”成为最大或最清晰的视觉主角。
- 周边能力、输入、接口、场景只能作为支撑，不得与中心主体同权。

沉淀建议：通用级。

## 5. 可复用版式规则

1. **阅读框架页**：不能只做目录导航，必须回答“高层应按什么顺序形成判断”。
2. **核心结论页**：用“主判断 + 3~4 个支撑判断 + 决策提示”，避免四卡平铺。
3. **框架说明页**：优先建立“输入—过程—输出—校验/回流”主链路，再放能力模块。
4. **分层架构页**：每层必须说明对上一层的支撑关系，不能只做能力清单。
5. **案例证明页**：左侧是判断链，右侧是证据链；图片不能替代论证。
6. **路线图页**：三阶段必须包含阶段门槛或验收产出，避免并列计划。
7. **决策建议页**：拍板事项必须独立突出，表格只承载责任、资源、时点、产出。
8. **总结页**：应输出一个可复用定义或最终判断框架，不只是四个关键词。

## 6. 可复用标题与表达规则

### 6.1 标题压缩规则

长标题应压缩为：

```text
不是 A，而是 B
核心不在 A，而在 B
要先完成 A，再推进 B
真正瓶颈不是 A，而是 B
从 A 走向 B，关键在 C
```

### 6.2 页面收口规则

- 底部结论必须提供增量：因此、所以、下一步、管理含义、拍板事项。
- 若底部结论只是复述标题，应删除或改成右侧洞察。
- 总结页应形成定义句：`X = 对象 + 能力 + 治理边界`。

沉淀建议：通用级。

## 7. 可复用图表语义规则

1. `comparison_table`：必须输出差异后的判断，不只是左右列举。
2. `swimlane_process`：必须有角色/阶段/动作/交付物，不应只是流程装饰。
3. `capability_map`：能力必须连接到场景或治理闭环，不只列能力名。
4. `value_chain_loop`：必须显式表达反馈方向和闭口机制。
5. `layered_stack_diagram`：每层要能解释“如何支撑上一层”。
6. `architecture_flow_diagram`：节点必须有真实流向，不同层级概念不得同级摆放。
7. `decision_table`：表格承载执行清单，拍板结论要独立呈现。
8. `case_gallery_cards`：案例只证明一个核心判断，避免素材墙。

## 8. 可复用页面节奏规则

建议本类技术高层汇报采用：

```text
总起定义 → 阅读框架 → 核心判断 → 问题定性 → 框架机制 → 能力/架构 → 案例证明 → 治理挑战 → 路线图 → 决策事项 → 最终定义
```

页面节奏控制：

- 前 3 页快速建立“为什么值得看”；
- 中段每 2~3 页必须有一个机制/闭环页，避免连续概念说明；
- 案例页只出现一次或少量出现，服务证明；
- 决策页必须从技术内容切回管理拍板；
- 收口页形成最终定义，不再展开新信息。

## 9. 不应沉淀的内容边界

以下内容不得进入长期通用规则：

1. 具体论文主题中的 AI/MBSE/GBDL/FMU/XAI 等专属术语；
2. 具体企业、案例、系统、生产线、装配示例；
3. 图片中的未核验数字、项目结论、产品能力表述；
4. 单页局部样式，如某一页的红条、图标、配色变体；
5. 只适用于 AI4MBSE 主题的章节结构；
6. 任何与无关主题绑定的 AI 算力、昇腾、NVIDIA 等术语。
